== OpenShiftプロジェクトの理解

プロジェクトは、デプロイメントを整理し管理するのに役立つOpenShiftの基本概念です。 OpenShiftプロジェクトは、ユーザーのグループ（または個人ユーザー）が、他のグループから隔離された状態でコンテンツを整理・管理することを可能にします。 各プロジェクトには、独自のリソース、ポリシー（誰がアクションを実行できるか、できないかを定義）、および制約（リソースのクォータや制限など）があります。 プロジェクトは、あなた（またはあなたのチーム）が作業に使用しているすべてのアプリケーションサービスとエンドポイントを「包むラッパー」のようなものだと考えてください。 内部的には、OpenShiftプロジェクトはKubernetesのnamespace（名前空間）の上に構築されていますが、プロジェクトという抽象化により、管理者には強化されたロールベースのアクセス制御（RBAC）などの追加機能が提供されます。

このセクションでは、Webコンソールとコマンドラインインターフェースの両方を使用してプロジェクトを操作する方法を探ります。 すべての用語を理解できなくても心配しないでください—これらの概念については、ワークショップ全体を通じて詳しく説明します。

== Webコンソールでのプロジェクトの探索

*{ocp4_starter_project}* という名前のプロジェクトがすでに作成されています。 まずは link:{ocp4_starter_console_url}[Webコンソール,window='_blank'] を通じてそれを探索することから始めましょう。

OpenShift Webコンソールにログインした後、プロジェクトのドロップダウンからあなたのプロジェクトを見つけて選択します。 プロジェクトの概要（Overview）を少し見てみましょう。ここには、プロジェクトのリソースとアクティビティの概要が表示されます。

=== Topology（トポロジー）ビュー

Topologyパースペクティブに慣れましょう。これは、プロジェクト内のアプリケーションとそれらのコンポーネントを視覚的に表現するものです。 左側のメニューから *Workloads*（ワークロード）に移動し、次に *Topology*（トポロジー）に移動します。

[.bordershadow]
image::openshift-console-topology.png[link="self",window=_blank]

現在は、まだ何もデプロイされていないため、アプリケーションやコンポーネントは表示されません。 ワークショップを進めていくと、この *Topology*（トポロジー）ビューでアプリケーションのコンポーネントを視覚化し、操作できるようになります。

== コマンドラインでのプロジェクトの操作

OpenShift CLIを使用してプロジェクトと対話することもできます。CLIは強力な自動化機能と高度な機能へのアクセスを提供します。 Webターミナルで、次のコマンドを使用してプロジェクトに切り替えます：

[source,role="copypaste",subs="attributes"]
----
oc project {ocp4_starter_project}
----

以下のような出力が表示されるはずです：

[.console-output]
[source,subs="attributes"]
----
Now using project "{ocp4_starter_project}" on server "{ocp4_starter_api_url}"
----

このコマンドは、割り当てられたプロジェクト内で作業するように現在のコンテキストを設定します。 以降のすべての `oc` コマンドは、特に指定しない限り、このプロジェクトのスコープ内で操作されます。

== 次のステップ

このワークショップ全体を通して、Webコンソールとコマンドラインインターフェースの両方を使用して、プロジェクト内でアプリケーションをデプロイおよび管理します。 Webコンソールは一般的なタスクのための直感的なビジュアルインターフェースを提供し、CLIはより高度な操作のための強力な自動化機能を提供します。 これで、プロジェクトに慣れ、両方のインターフェースからアクセスする方法がわかったので、アプリケーションのデプロイを開始する準備ができました！
